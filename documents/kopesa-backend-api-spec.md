# Kopesa Backend — Technical API Specification

**Target stack:** Go 1.22+ · Fiber v2 · Cassandra (ScyllaDB compatible) · Kafka · DDD + CQRS-lite
**Audience:** AI / engineering team implementing the backend for the Kopesa field-lending prototype
**Source of truth for shapes:** the React/TypeScript prototype (`src/types/index.ts`, `src/data/*`, `src/pages/*`, `src/store/slices/*`)
**Version:** 1.0 — 2026-04-23

---

## 1. Architecture Overview

### 1.1 Bounded Contexts (DDD)
Each context owns its aggregates, Cassandra keyspace, and Kafka topics. They communicate **only** via domain events on Kafka — never direct DB reads across contexts.

| Context | Aggregates | Purpose |
|---|---|---|
| **iam** | User, Role, Branch, Area, Subscription | Auth, RBAC, org structure, role-based subscription tiers |
| **loans** | LoanApplication, Loan, Disbursement | Origination → assessment → approval → disbursement → servicing |
| **arrears** | ArrearsCase, PTP, ArrearsPayment | Collections workflow, allocation, PTP, escalation, write-off |
| **campaigns** | Campaign, CampaignRoute, RouteStop, Lead | Field campaigns, route planning, lead capture & qualification |
| **messaging** | MessageTemplate, MessageLogEntry | SMS/WhatsApp via provider gateway |
| **attachments** | Attachment | Evidence files (POP, KYC, loan docs) — S3-compatible object store |
| **reporting** | (read models only) | Executive & operations dashboards (materialized via Kafka consumers) |
| **audit** | AuditEvent | Immutable audit log of every maker/checker action |

### 1.2 Layered Layout (per context)
```
internal/<context>/
  domain/        # entities, value objects, domain events, repository interfaces
  application/   # use-cases (command + query handlers), DTOs
  infrastructure/
    cassandra/   # gocql repository implementations
    kafka/       # producers + consumers
    http/        # Fiber handlers + DTO mapping
  ports/         # interface definitions (hexagonal)
```

### 1.3 Cross-cutting
- **Router:** `gofiber/fiber/v2`
- **Cassandra driver:** `gocql/gocql` with `scylladb/gocqlx/v2` for query building
- **Kafka:** `segmentio/kafka-go` (producer + consumer groups)
- **Auth:** JWT (HS256 for service-to-service, RS256 for user tokens), 15-min access + 30-day refresh
- **Validation:** `go-playground/validator/v10`
- **Migrations:** `golang-migrate` with CQL files
- **Observability:** OpenTelemetry traces, Prometheus metrics, structured zap logs
- **Idempotency:** all `POST` mutating endpoints accept `Idempotency-Key` header; cached 24h in `iam.idempotency_keys`

### 1.4 Request/Response Envelope
All JSON responses use this envelope:
```json
{
  "data":  { /* resource or list */ },
  "meta":  { "page": 1, "pageSize": 25, "total": 142, "requestId": "req_..." },
  "error": null
}
```
Error responses:
```json
{
  "data": null,
  "meta": { "requestId": "req_..." },
  "error": {
    "code": "ARREARS_PTP_INVALID_DATE",
    "message": "Promised date must be in the future",
    "details": [{ "field": "promisedAt", "rule": "future" }]
  }
}
```

### 1.5 Common Headers
| Header | Direction | Notes |
|---|---|---|
| `Authorization: Bearer <jwt>` | in | required on all `/v1/*` except `/v1/auth/*` |
| `X-Active-Role` | in | role the user is currently acting as (must be in user's allowed roles) |
| `X-Branch-Id` / `X-Area-Id` | in | scope override; server validates against role scope |
| `Idempotency-Key` | in | UUID, required on financial mutations |
| `X-Request-Id` | out | echoed back in `meta.requestId` |

## 2. Conventions

### 2.1 IDs
- All IDs are **TimeUUID v1** (Cassandra-friendly, sortable). Exposed as strings.
- Loans keep a human reference: `LN-44021` (`reference` field), generated by `loans.reference_seq` counter table.
- Branches keep a `code` (e.g. `JHB01`) used for CSV imports — see `arrears.import_cases`.

### 2.2 Money
- All amounts are **integer minor units** (cents) over the wire: `2450` = R 24.50.
- Currency is implicit `ZAR` v1; field `currency` reserved for v2.

### 2.3 Timestamps
- ISO-8601 UTC strings, e.g. `2026-04-23T08:15:00Z`.
- Cassandra stores as `timestamp`.

### 2.4 Pagination
Cursor-based for hot lists, offset for admin tables.
- Cursor: `?cursor=<opaque>&limit=25` → response `meta.nextCursor`
- Offset: `?page=1&pageSize=25` → response `meta.total`

### 2.5 Filtering & Sorting
- `?filter[status]=active&filter[branchId]=br-jhb-cbd`
- `?sort=-daysPastDue,clientName` (`-` = desc)

## 5. loans

### 5.1 Aggregate Cassandra Schema (illustrative)
```cql
CREATE TABLE loans.loans (
    id          uuid,
    reference   text,            -- "LN-44021"
    branch_id   uuid,
    client_name text, client_phone text,
    principal   bigint, term_months int, rate_bp int,   -- rate stored as basis points
    status      text,
    outstanding_balance bigint,
    next_due_date date,
    created_by  uuid, assessed_by uuid, approved_by uuid, disbursed_by uuid,
    created_at  timestamp, updated_at timestamp,
    PRIMARY KEY (id)
);
CREATE TABLE loans.loans_by_branch (branch_id uuid, status text, id uuid, ...
    PRIMARY KEY ((branch_id), status, id)) WITH CLUSTERING ORDER BY (status ASC, id DESC);
CREATE TABLE loans.applications (id uuid, loan_id uuid, ..., PRIMARY KEY (id));
```

## 15. Non-Functional Requirements

- **Latency:** p95 < 200ms for read endpoints, < 400ms for write.
- **Throughput target (v1):** 500 RPS sustained, 2k RPS burst.
- **Availability:** 99.9% per service.
- **Multi-tenant ready:** every table includes implicit `tenant_id` column (default `kopesa`); add to PK in v2.
- **Offline-friendly clients:** all `POST` mutations idempotent; messaging & attachments tolerate retries.
- **Localization:** all user-facing strings server-side returned in English; i18n keys reserved for v2.
